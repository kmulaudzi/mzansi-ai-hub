# Release 005 — Embedding Engine

## Overview

Release 005 introduces the **Embedding Engine**, the first Artificial Intelligence capability of the Heritage Intelligence Engine.

Previous releases focused on building the software architecture required to ingest, search and process heritage knowledge.

This release teaches the platform how to convert text into numerical vector representations (embeddings), allowing the system to understand semantic meaning rather than only matching keywords.

Although Release 005 still performs keyword-based retrieval, every heritage document chunk is now represented as a high-dimensional embedding generated by a Hugging Face Sentence Transformer model.

This release lays the foundation for semantic search in Release 006.

---

# Objectives

- Introduce the Embedding Engine.
- Learn how Hugging Face embedding models integrate into our architecture.
- Generate vector embeddings for every document chunk.
- Cache embedded chunks during application startup.
- Separate expensive preprocessing from user searches.
- Improve the Gradio runtime experience with progress reporting.
- Document the complete runtime architecture.

# Architectural Decisions

## Why create an Embedding Engine?

Although Hugging Face already provides embedding models, the Heritage Intelligence Engine requires its own architectural layer.

The Embedding Engine:

- hides the Hugging Face implementation
- standardises embedding generation
- keeps business logic independent of external libraries
- allows future models to be swapped without changing the rest of the platform

---

## Why cache embeddings?

Generating embeddings is computationally expensive.

Instead of embedding documents every time a user performs a search, Release 005 prepares the knowledge once during application startup.

```
Application Startup

↓

Load Documents

↓

Create Chunks

↓

Generate Embeddings

↓

Cache

↓

Search Many Times
```

This dramatically improves runtime performance.

---

# Current Limitation

Although every chunk now contains an embedding, searches still rely on keyword matching.

Current flow

```
User Query

↓

Keyword Search

↓

Ranking
```

The embeddings are generated and stored but are not yet used to retrieve documents.

This limitation is intentional and prepares the architecture for Release 006.

---

# Lessons Learned

Release 005 introduced several important engineering concepts.

- Embeddings represent semantic meaning.
- AI models should be separated from business architecture.
- Expensive preprocessing should happen once.
- Runtime startup is different from search-time execution.
- Gradio should remain a presentation layer.
- `app.py` serves as the runtime blueprint of the application.
